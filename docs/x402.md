# x402 micropayments (`cambrian pay`)

Opt-in pay-per-call against Cambrian's x402 gateway, as an alternative to an API
key. **Spends real USDC on Base.** Strictly additive: the default API-key paths
and the zero-runtime-dependency core are untouched — the payment SDK is loaded
lazily and is never bundled.

## Verified live (2026-06-17)

- **Gateway:** `https://x402.cambrian.network`. `GET /health` reports the
  payment network and price, currently Base USDC at `$0.05` per call.
- **Payable paths:** the same `/api/v1/<group>/<resource>` the regular API uses —
  **all groups** (`solana`, `evm`, `deep42`, `risk`). Examples:
  `/api/v1/evm/price-current`, `/api/v1/deep42/social-data/alpha-tweet-detection`,
  `/api/v1/perp-risk-engine`.
- **Protocol:** x402 **v2**, scheme `exact`, EIP-3009 `TransferWithAuthorization`
  over **USDC on Base mainnet** (`eip155:8453`), **$0.05/call**, facilitator-settled
  (the payer pays no gas). Confirmed with real on-chain payments via the CLI
  (USDC balance decremented; settlement tx in the `payment-response` header).

> **Why the official SDK, not a hand-rolled payload.** The gateway emits a v2
> `accepts[]` (`amount`, CAIP-2 `network`, top-level `resource`) that the legacy
> `x402`/`x402-fetch` (v1) packages reject. The **scoped `@x402/*` v2 packages** —
> the same family the server runs (`@x402/core` + `@x402/evm`) — are wire-compatible.
> We use them for signing/settlement and only add our own guardrails around them.

## Flow

1. The CLI sends an unpaid pre-flight `GET` to learn the price.
2. Enforce the spend cap (`--max-amount`, default $0.10) and print a cost preview.
3. Require `--yes`; without it we stop **before** any signing/spend.
4. Hand off to `@x402/fetch`'s `wrapFetchWithPayment(fetch, client)` (with an
   `ExactEvmScheme(viemSigner)` registered for `eip155:*` and a cap `PaymentPolicy`),
   which signs the EIP-3009 authorization and retries with the payment header.
5. On `200`, print the data; the settlement receipt goes to stderr.

## Guardrails (a tool that spends money)

- **Spend cap** `--max-amount <usd>` (default `0.10`) — enforced twice: by our
  pre-flight check and by an SDK `PaymentPolicy` that filters out over-cap options.
- **Explicit authorization** — `--yes` required; otherwise preview-only.
- Network/asset/payTo shown before paying.
- Wallet key read **only** from `CAMBRIAN_X402_PRIVATE_KEY` at runtime — never
  persisted, logged, or written to config.

## Dependencies & packaging

- `viem` + `@x402/core` + `@x402/evm` + `@x402/fetch` are **devDependencies**, marked
  **external** in the esbuild bundle, and lazy-`import()`ed only by `cambrian pay`.
  `npm i -g cambrian` stays **zero runtime dependencies** (`dist/cli.js` contains
  only the `import("…")` specifiers, no inlined library code).
- If the SDK isn't installed, `cambrian pay` prints an install hint and exits:
  `npm install -g @x402/fetch @x402/evm viem`.

## Usage

```bash
export CAMBRIAN_X402_PRIVATE_KEY=0x<funded-base-mainnet-key>
npm install -g @x402/fetch @x402/evm viem          # one-time, the lazy peers

cambrian pay deep42 social-data/alpha-tweet-detection --limit 1 --yes
cambrian pay base price-current --token-address 0x4200…0006 --yes --output table
cambrian pay deep42 social-data/alpha-tweet-detection --limit 1 --fields tokenSymbol,text --yes
```

Without `--yes` you get a price preview and nothing is spent.

## Not in scope (yet)

Solana-network payments (the signer registers `eip155:*` / EVM only), CDP-account
or OS-keychain key custody, and multi-`accepts` selection.
